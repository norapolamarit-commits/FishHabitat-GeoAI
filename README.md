# AI-Based Fishing Habitat Suitability Assessment

A GeoAI research platform integrating satellite remote sensing, oceanographic
data, climate reanalysis, and explainable AI to assess fishing habitat
suitability in the Gulf of Thailand and Andaman Sea.

## Honesty / scope

This is a research prototype built entirely on **free, no-authentication
public data sources** (NOAA CoastWatch ERDDAP, NASA GIBS/Worldview,
Open-Meteo, World Bank, Marine Regions). There is no real CPUE (catch per
unit effort) dataset for this region publicly available, so the model is
trained against a **literature-derived Habitat Suitability Index proxy**
(see `apps/api/pipeline/hsi_formula.py` for citations) — not real fishing
outcomes. Every prediction the API returns carries an explicit `data_basis`
field (`real_observed`, `real_derived`, or `proxy_label`) and the UI shows
this transparently rather than presenting proxy figures as validated fact.

## Architecture

```
apps/
├── web/    Next.js 15 + TypeScript frontend (App Router, Tailwind, shadcn/ui,
│           Framer Motion, Leaflet, Chart.js, React Flow)
└── api/    FastAPI backend
    ├── pipeline/   real data ingestion + ML training (run offline/on-demand)
    ├── routers/    REST API served to the frontend
    └── models/     trained model artifacts + metrics (generated by the pipeline)
```

## Running it

### 1. Backend

```bash
cd apps/api
python3.12 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# One-time (or whenever you want to refresh the real data + retrain):
python -m pipeline.train_ensemble

# Start the API:
uvicorn main:app --reload --port 8000
```

The training step fetches real data from public APIs (SST, chlorophyll,
bathymetry, salinity, wind, wave, current, satellite imagery), trains the
stacking ensemble, and writes artifacts to `apps/api/models/`. It takes a
few minutes on first run (subsequent runs reuse the on-disk cache in
`apps/api/cache/`). The API will return 503s on endpoints that need these
artifacts until this has run at least once.

### 2. Frontend

```bash
cd apps/web
npm install
npm run dev
```

Open `http://localhost:3000`. The frontend proxies `/api/*` to
`http://localhost:8000` (configurable via `NEXT_PUBLIC_API_URL`).

### 3. Optional: chatbot (Pathumma / AI FOR THAI)

The `/chatbot` page is fully usable out of the box: without an API key, it
answers by matching your question against a real TF-IDF retrieval index over
the platform's own documentation (`apps/api/chat_knowledge.py`) — never a
canned or fabricated reply, just the closest real doc chunk, clearly labeled
"Matched from documentation". To enable fuller, generated answers instead:

1. Register for a free API key at https://aiforthai.in.th
2. `export AIFORTHAI_APIKEY=your-key-here` before starting `uvicorn` (or add
   it to a `.env` file the backend loads)

Each response is tagged `mode: "llm"` or `"retrieval"` so the UI is always
honest about which one answered.

## Deploying (Render)

`render.yaml` at the repo root defines both services as a Render "Blueprint":

- **fishing-habitat-api** — Python web service, root `apps/api`, runs
  `uvicorn main:app --host 0.0.0.0 --port $PORT`. `apps/api/models/` is
  committed (not gitignored) specifically so this works with zero build-time
  training — no internet-dependent pipeline run needed on deploy.
- **fishing-habitat-web** — Node web service, root `apps/web`, runs
  `npm run build` then `npm start`.

To deploy:

1. Push this repo to GitHub.
2. In Render, **New → Blueprint**, point it at the repo. It reads
   `render.yaml` and creates both services.
3. Set env vars Render can't infer on its own:
   - `AIFORTHAI_APIKEY` on the API service (optional — leave blank for the
     documentation-retrieval chatbot fallback).
   - `NEXT_PUBLIC_API_URL` on the web service — the API service's URL once
     Render assigns it (e.g. `https://fishing-habitat-api-xxxx.onrender.com`).
     You'll need to create the API service first, copy its URL, then set this
     on the web service and redeploy.
   - `JWT_SECRET_KEY` is auto-generated by the blueprint — no action needed.

Two things to know about Render's free/starter tier:

- **The SQLite database resets on every deploy** — Render's default disk is
  ephemeral, so user accounts (which are opt-in and non-critical to the rest
  of the site) don't persist across redeploys. Add a Render persistent disk
  mounted at `apps/api/data` if you want accounts to survive deploys.
- **Cold starts**: free-tier services sleep after inactivity; the first
  request after a while will be slow while it wakes up.

`.env.example` in `apps/api/` documents the same two backend env vars for
local use.

## Accounts

Signing in is **optional** — every page works without an account. Creating
one (`/login`) just unlocks saving your home location + target species
(used to personalize Map/Prediction defaults) and persistent chatbot history.
Auth is a real SQLite-backed system (bcrypt password hashing, JWT in an
httpOnly cookie) — no third-party OAuth needed. The database file is created
automatically at `apps/api/data/app.db` on first run (gitignored — it's
per-deployment state, not source).

**Before deploying anywhere beyond your own machine**, set `JWT_SECRET_KEY`
to a random value (`python -c "import secrets; print(secrets.token_hex(32))"`).
`auth.py` falls back to a fixed dev-only string otherwise, which is fine for
local demos but not for anything reachable by other people.

## Notes for anyone extending this

- `pipeline/features.py` is the single source of truth for feature
  extraction — imported by both training and serving so they can't drift
  apart.
- Import order matters in any process touching both torch and xgboost/shap:
  see the comment block at the top of `pipeline/train_ensemble.py` and
  `main.py` — a real macOS OpenMP conflict between the two libraries
  segfaults the process otherwise.
- The `config.DATASET_CATALOG` in `apps/api/config.py` is the single source
  of truth for the Datasets page — it honestly marks which of the 19
  datasets in the original project spec are actually wired in (`live`) vs.
  require credentials this build doesn't have (`not_wired`, e.g. Sentinel
  Hub, SAR, THEOS-2).
- Password hashing uses the `bcrypt` library directly (`auth.py`), not
  `passlib` — passlib 1.7.4's internal self-test crashes against
  `bcrypt>=4.1`'s stricter API. If you see `ValueError: password cannot be
  longer than 72 bytes` from `passlib`, that's why; this project sidesteps
  it entirely rather than pinning an old bcrypt.
- Pydantic's `EmailStr` needs the separate `email-validator` package
  (in `requirements.txt`) or the API fails to start with a `ModuleNotFoundError`
  buried inside a Pydantic schema-generation traceback.
